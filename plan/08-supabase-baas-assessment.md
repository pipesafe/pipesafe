# 08 — Supabase, BaaS, and the MongoDB Data-Plane Void

Supabase ($10.5B valuation, ~$170M ARR) proved that a database wrapped in one SDK, one gateway, and one security model is a generational business — and MongoDB proved the demand on its own platform, then abandoned it: Atlas App Services went dark on September 30, 2025, and nobody has claimed the typed data plane it vacated. This assessment (evidence: the two Supabase research briefs, 2026-07-05) concludes PipeSafe should **not** become a BaaS, but should grow three BaaS-shaped edges on the existing roadmap: **agent-channel assets now** (EPIC-K, P1), **a typed realtime SDK over change streams next** (EPIC-L, P2), and — as a firm strategic decision, not a demand-gated option — **out-of-the-box RLS for MongoDB plus a self-hostable typed data API gateway** (EPIC-M, P3). The last is the maintainer's key insight: out-of-the-box RLS is the main reason teams cannot move from Supabase to Atlas + PipeSafe today. RLS is the migration unlock, not a nice-to-have.

## 1. What Supabase is: two moats, neither of them components

**The composition thesis.** The self-hosted product is one docker-compose file wiring eleven containers around a single Postgres instance, and remarkably little of the request path is original: PostgREST (adopted), GoTrue auth (forked from Netlify), Kong gateway (adopted), imgproxy (adopted); the genuinely built pieces are the Elixir realtime server, storage, Studio, and Supavisor. Every component is individually replaceable and individually unremarkable. The product is the integration: one shared database as coordination point, one JWT as universal identity, one gateway URL, one client SDK, one dashboard.

**The technical moat is RLS.** PostgREST validates the JWT, switches the Postgres role, and exposes claims — then Row Level Security policies (`auth.uid() = user_id`) enforce authorization on _every_ access path uniformly: REST/GraphQL queries, storage objects, realtime change subscriptions (re-checked per subscriber), even Broadcast/Presence (authorized via a rolled-back probe query against a policies table). One policy language, written once, enforced everywhere, with zero authz code in any service. This composability is what every "Supabase for X" must reproduce, and the hardest thing to reproduce on MongoDB — because MongoDB has no RLS (§4).

**The business moat is the agent channel.** The Series F ($500M at $10.5B, June 2026 — [CNBC](https://www.cnbc.com/2026/06/04/database-startup-supabase-raises-500-million-10point5-billion-valuation.html)) priced ~$170M ARR ([Sacra](https://sacra.com/c/supabase/)) at a ~60x multiple because **AI agents now deploy the majority of new Supabase databases, with Claude Code the single largest contributor since the start of 2026**, and "Supabase for Platforms" grew 370% in six months ([MLQ Series F summary](https://mlq.ai/news/supabase-raises-500m-series-f-at-105b-valuation-doubles-in-8-months/)). Lovable auto-provisions a Supabase backend per workspace; Neon's $1B Databricks exit cited 80% agent-provisioned databases. The clarifying comparison is **Convex**: the closest DX cousin — reactive-by-default typed TS queries over a document-relational model, arguably a tighter developer experience than Supabase's — valued at roughly 1/200th of Supabase. **Channel beats DX.** Supabase courts the channel deliberately: an official [MCP server](https://supabase.com/docs/guides/ai-tools/mcp), [Agent Skills](https://supabase.com/blog/supabase-agent-skills), llms.txt docs, and a free tier that functions as CAC for agent-built apps.

## 2. The MongoDB BaaS void

MongoDB dismantled its entire BaaS in two waves: March 2024 (Atlas GraphQL API and Static Hosting deprecated — [forum notice](https://www.mongodb.com/community/forums/t/deprecating-mongodb-atlas-graphql-and-hosting-services/283080)), September 2024 (Device Sync, the Realm SDKs, the Data API, Custom HTTPS Endpoints — [EOL notice](https://www.mongodb.com/community/forums/t/atlas-device-sync-end-of-life-and-deprecation/296687)), with the whole App Services platform going dark on **September 30, 2025**. Only Triggers survived, re-homed in the Atlas UI. It was the third iteration of the same product (Stitch → Realm/App Services → gone); official guidance to users was essentially _go build your own backend_, and the community reaction — "rug pulled" — left real reputational residue.

Who is filling it? **Appwrite × MongoDB (April 2026)** is the most important datapoint: an official partnership shipping MongoDB support in self-hosted Appwrite ([announcement](https://dev.to/appwrite/april-product-update-mongodb-support-appwrite-190-realtime-upgrades-and-ai-tooling-1eg6)) — proof MongoDB Inc. wants the platform layer to exist as a partner's business. But it is self-host-only and puts Appwrite's own abstraction over Mongo rather than a Mongo-native typed API. Below that: small API shims (RESTHeart, DreamFactory, Modelence) with no escape velocity, while Convex absorbs document-model-plus-types demand without MongoDB at all. **Nobody owns the typed data plane** — the layer where policies, subscriptions, and APIs are expressed in the application's own types and compiled to Mongo primitives. That is the lane this plan already occupies.

## 3. Where PipeSafe is ahead — and what a BaaS needs that it lacks

**Ahead on typing, decisively.** supabase-js's celebrated inference is a compile-time parser for `select()` strings; its _filter_ side is weak: dotted paths fall through to raw `string`, and every comparison operator keeps an untyped escape overload (`PostgrestFilterBuilder.ts:317-318` — `gt(column: string, value: unknown)`), so any typo'd column compiles. There is no filter-driven narrowing of result types, and errors surface as `ParserError<...>` strings. PipeSafe's recursive path validation, `PipeSafeError` operand brands, and union narrowing through stages are categorically stronger — and work without a codegen step or the schema-drift problem of `supabase gen types`.

**Ahead on expressiveness.** Supabase's query builder tops out at filters, FK-embedded reads, and five implicit aggregates; anything real — grouping, window functions, arbitrary joins, multi-step transforms — means writing a SQL view or function and calling `.rpc()`. Every non-trivial Supabase app accumulates a pile of untyped SQL. That is exactly where a typed aggregation pipeline is _more_ expressive in the client language, and manifold's DAG materialization has no Supabase equivalent at all.

**Missing: the entire platform.** Auth and JWT issuance, storage, a functions runtime, Studio, hosting, billing, provisioning, abuse prevention, and a loss-making free tier. These are platform economics — Supabase raised >$1B; Appwrite and Nhost raised tens of millions for weaker versions — and a pre-revenue library company cannot cross that chasm. There is also a structural landmine: **MongoDB's SSPL prohibits offering MongoDB itself as a service**, so anything hosted must be BYO-Atlas, formally partnered, or built on a Postgres-backed Mongo-compatible layer (conceding the argument). **Full BaaS is explicitly out of scope.** What is in scope is the one differentiated ingredient a Mongo BaaS would need anyway: a typed pipeline compiler through which policies, subscriptions, and APIs all compile to typed pipelines.

## 4. The three wedges

**Wedge 1 — Agent-channel assets (EPIC-K, prefix AGENT, P1: near-term, near-zero marginal cost).** A PipeSafe MCP server plus SKILL.md agent skills and llms.txt docs, so coding agents generate _correct, typed_ PipeSafe code instead of stringly `aggregateRaw`. Supabase's growth is agent-driven and Claude Code is its largest deployer; Prisma Next already ships [SKILL.md skills](https://www.prisma.io/docs/ai/tools/skills) — this is table stakes for developer-tool distribution now. PipeSafe does not need to own the channel; it needs to be the thing agents reach for when the project is already MongoDB. The `PipeSafeError` message grammar is already structured input for agents. Pure accelerant for the existing library strategy, compounding regardless of any other decision.

**Wedge 2 — Typed realtime SDK over change streams (EPIC-L, prefix RT, P2).** "Typed live queries for MongoDB," attacking Supabase's _weakest_ pillar with MongoDB's _strongest_ uncontested primitive. Supabase Realtime polls a logical replication slot via wal2json, re-checks RLS **per subscriber per event** (their docs: 100 subscribers to one insert = 100 reads), processes changes on a single thread ("compute upgrades don't have a large effect"), documents delivery as best-effort, drops events across reconnects ([GitHub #5641](https://github.com/orgs/supabase/discussions/5641)), and caps Pro at 500 concurrent connections. Change streams are structurally better: resumable via tokens, **server-side filterable with an aggregation pipeline** (per-subscriber predicates push down into the stream instead of fanning out as re-checks), and concurrently consumable with no slot bookkeeping. EPIC-J's executed CDC spike already solved the hard parts — pre-snapshot `operationTime` ordering, the resume-token failure taxonomy (codes 286/280/50811), the silent-idle watchdog. EPIC-L builds the client-facing half on that machinery.

**Wedge 3 — Out-of-the-box RLS for MongoDB + self-host typed data API gateway (EPIC-M, prefix RLS, P3 — firmly on the roadmap, further down the line, not demand-gated).** This is the maintainer's strategic decision, recorded here as such: **out-of-the-box row-level security is the main reason teams cannot move from Supabase to Atlas + PipeSafe — it is the migration unlock, not a nice-to-have.** MongoDB has no RLS: `$$USER_ROLES` is coarse and view-shaped, `$redact` is per-query, App Services' rules engine is dead. The design direction: typed policies — a policy is a typed `$match`/`$expr` fragment against the schema — compiled and injected into every read and write via EPIC-D's interceptors (library enforcement) and/or a deployable gateway (a "PostgREST for MongoDB, but typed end-to-end" enforcement point), and compiled into change-stream filter pipelines so subscriptions are policy-filtered by composition with EPIC-L. Authentication is **never built**: auth arrives via adapters (Better Auth, Clerk, generic JWT claims — all have Mongo-ecosystem support), and policies consume verified claims. Everything else Supabase bundles — storage, functions, Studio, hosting — stays out of scope per §3's platform economics and the SSPL constraint.

## 5. The Supabase → Atlas + PipeSafe migration play

The target persona: teams that outgrew Supabase's realtime (connection caps, silently dropped events, graduating to Ably/Pusher) or got burned by RLS footguns — `security definer` views silently bypassing policies (a repeated source of real data leaks), undebuggable generic "permission denied" failures, per-row policy costs — or simply want the document model without Convex's proprietary runtime. The recurring Supabase complaint pattern is **leaky abstraction**: Postgres is hidden until it breaks, then you must be a Postgres DBA. PipeSafe's counter-position is a typed layer that makes the database _legible_ rather than hidden: policies that are compile-checked TypeScript against your schema instead of SQL strings debugged through nine log tabs; live queries with resume semantics instead of best-effort channels; typed pipelines plus manifold's materializations instead of a pile of SQL functions. Without EPIC-M the pitch dies at "how do I do RLS?"; with it, the migration story is complete: auth adapter + typed policies + typed live queries + typed pipelines, on Atlas.

## 6. Honest counter-arguments (preserved, not litigated away)

- **The gap may exist because demand was weak.** MongoDB retired Stitch/Realm/App Services with perfect usage visibility, citing focus and low adoption. The wedges hedge this by being roadmap-aligned rather than bet-the-company.
- **The BaaS buyer mostly isn't the MongoDB buyer.** Frontend/no-backend developers pick Supabase; Mongo's base is backend teams who write Express APIs without pain. EPIC-M targets the _migration_ persona, not the greenfield BaaS buyer.
- **Postgres won this era** — Supabase at $10.5B, Neon's exit, Firebase adding Postgres, pgvector — and Convex absorbs document-model demand at ~$250M-class scale without Mongo.
- **The agent channel compounds against latecomers**: every month of agent output adds Supabase-shaped code to future training corpora. (This is the argument for EPIC-K's P1 timing.)
- **Full-BaaS revisit criteria stand**: only with evidence from the three wedges, dedicated funding, and a MongoDB Inc. relationship — the demand MongoDB's own telemetry apparently didn't show must be demonstrated, not assumed.

## 7. The MongoDB Inc. partnership angle

MongoDB killed its own BaaS, then blessed two partners in adjacent lanes within 18 months: Appwrite for the platform layer, Prisma for the typed client layer. The read is unambiguous — MongoDB wants the DX layer to exist but refuses to own it. The lane neither partner covers is exactly this document's subject: **the typed pipeline / realtime / RLS data plane** — Appwrite abstracts Mongo away rather than typing it; Prisma Next has no realtime, no policy engine, and no DAG. That is the partnership pitch, an extension of plan/06 §3's "partner where cheap, differentiate where it counts." One standing caveat: Prisma Next is simultaneously the anointed partner in the nearest lane and PipeSafe's chief competitor (plan/00's window logic) — a partnership conversation strengthens the wedge positioning but must never gate it.
