# EPIC-M — RLS Policy Engine (TRD)

## Overview

**Why this epic is the migration unlock.** Out-of-the-box row-level security is _the_ single reason teams cannot leave Supabase for Atlas + PipeSafe. RLS is Supabase's technical moat: one policy language, written once, enforced uniformly across every access surface — REST, GraphQL, Storage, and Realtime all consult the same Postgres policies via `auth.uid()` and the JWT-derived role, with **zero authz code in any service** (see `../08-supabase-baas-assessment.md` and the architecture brief §4). MongoDB has no equivalent. There is no per-row policy the server evaluates; `$$USER_ROLES` (7.0+) is coarse and view-only, `$redact` is per-query not ambient, and Atlas App Services — the one product that gave MongoDB RLS-shaped rules in a hosted gateway — was **EOL'd September 2025** with the official guidance "build your own backend." The market slot is vacant and demand is proven.

This epic builds the substitute: **declarative, compile-time-typed policies that compile into `$match`/filter predicates injected at every data-access seam** — reads, writes, aggregations, and change streams. It is the load-bearing wall of any "Supabase for MongoDB" story. It is a firm, committed roadmap item at **P3** — sequenced after the P0–P2 core (typed CRUD, schema, hooks, incremental, connectors) but _not_ demand-gated: without it the BaaS-shaped pieces (EPIC-L realtime, a future data-API gateway) do not compose the way Supabase's do.

**The PipeSafe differentiator.** Supabase policies are SQL strings — unchecked until they run, and their failures are the community's #1 footgun (silent `permission denied`, no explanation; see the market brief §3). PipeSafe policies are **typed predicates over the collection `Schema` plus a typed `claims` context**, validated at compile time against the schema by the same machinery that powers `$match` (`ResolveMatchOutput`, `PipeSafeError` operand brands). A typo'd field in a policy is a compile error, not a production data leak. And they compile to the same typed pipeline substrate everything else in PipeSafe already produces.

**Scope.** Typed policy definition API; a compiler from policy → find-filter / aggregate-`$match` / update-and-delete-filter / insert-`WITH CHECK` injection, riding EPIC-D interceptors as the enforcement substrate; recursive `$lookup`/`$unionWith`/`$facet` sub-pipeline traversal (the key spike finding); auth adapters for JWT claims (Better Auth / Clerk / custom) with the claims schema typed via EPIC-C; change-stream/realtime composition (the EPIC-L seam); fail-closed defaults with an explicit service-role escape hatch; policy-denial observability into the EPIC-G event log; policy-testing utilities tied to EPIC-H; and a Supabase-migration mapping guide.

**Out of scope.** Building auth (issuance, sessions, OAuth) — PipeSafe integrates adapters, never issues tokens (market brief §6: "integrate, never build auth"). The HTTP/WebSocket enforcement _server_ itself (the gateway packaging decision is scoped here, but the transport, routing, and hosting belong to a future BaaS epic and `plan/06`). Field-level redaction beyond document-level policies (a follow-up). `$$USER_ROLES`-based view enforcement (a complementary server-side primitive, noted but not the primary mechanism).

**Plan docs:** `../08-supabase-baas-assessment.md` (the BaaS gap and RLS-substitute framing) · `../06-architecture-packaging-licensing.md` §5 (the enforcement-point / gateway open decision).
**Related epics:** **EPIC-D** (hooks & transactions) — the typed interceptor API is the enforcement substrate this epic compiles into; **EPIC-C** (runtime schema validation) — Standard Schema types the claims context and the policy predicates; **EPIC-L** (realtime) — change-stream subscription authz composes the same compiled predicates; **EPIC-G** (event log/executor) — policy-denial events; **EPIC-H** (testing framework) — assert-visibility fixtures reuse the data-test harness.
**Spike:** `../spikes/rls-policy-injection.spike.ts` (EXECUTED, this TRD).

## Spike findings

`rls-policy-injection.spike.ts` ran against a single-node `MongoMemoryReplSet` (mongodb-memory-server 10.2.x, driver 6.x). All checks pass. It seeds a multi-tenant `docs` collection and an owner-scoped `secrets` collection, registers per-collection read/write/check policies, and exercises three principals (a tenant-1 user, a tenant-1 admin, a tenant-2 user) plus a service context.

- **Reads with injected predicates return exactly the permitted rows.** Each principal's `find` ANDs the compiled read filter with the user filter; the three principals saw disjoint, correct row sets, and a user filter could only _narrow_ (never widen) the policy. This is the easy, uncontroversial half.

- **THE key finding — root-level `$match` prepend is NOT enforcement; `$lookup` is a wide-open bypass.** With only the _root_ collection's policy prepended, a malicious aggregation read another collection wholesale: `$lookup: { from: "secrets", pipeline: [{ $match: {} }] }` returned tenant-2 secrets to a tenant-1 user. Closing the hole requires **recursive traversal**: the compiler must prepend the _foreign_ collection's read policy into every `$lookup`/`$unionWith` sub-pipeline, resolved against the same claims — and recurse into nested `$lookup`-inside-`$lookup` and `$facet` branches. Verified leak → verified closed. Corollaries the spike pinned empirically:
  - **Shorthand `$lookup`** (`localField`/`foreignField`, no pipeline) accepts a _coexisting_ injected `pipeline: [{ $match: policy }]` (MongoDB ≥5.0 concise-correlated-subquery form); the policy ANDs with the equality join, so shorthand lookups are closable without rewriting to `let`/pipeline form.
  - **Bare-string `$unionWith`** (`{ $unionWith: "secrets" }`) leaks and must be rewritten to object form with an injected pipeline.
  - **`$graphLookup` has no sub-pipeline and therefore no injection point** — a literal `startWith` pulled tenant-2 docs across the boundary. It cannot be guarded and must be **denied** on any policy-bearing collection. Same for `$out`/`$merge` appearing in a read path (write-through bypass) — denied.

- **Writes: policy ANDed in gives RLS-equivalent semantics.** Compiling the write policy as `$and: [policyFilter, userFilter]` made a cross-tenant `updateOne`/`deleteOne` report `matchedCount`/`deletedCount` 0 — the row is _invisible_, not an error, matching Postgres RLS. Insert `WITH CHECK` is client-side predicate evaluation before the driver call: a cross-tenant insert threw a typed `PolicyDeniedError` and wrote nothing. Update `WITH CHECK` needs `$set` inspection to stop privilege-escalation via `$set: { tenantId: <other> }` — verified for `$set`; full post-image validation (pipeline updates, `$rename`) is flagged as a ticket.

- **`$expr` claim injection is real and non-obvious.** With an attacker-controlled claim string `sub = "$tenantId"`, `{ $expr: { $eq: ["$ownerId", claims.sub] } }` interpreted the claim as a **field path** and matched a doc where `ownerId === tenantId`. Passing values as BSON is _necessary but not sufficient_ inside `$expr`. Two safe forms: compile to **query-context filters** by default (`{ ownerId: claims.sub }` stays literal), and if `$expr` is unavoidable, wrap every claim value in `{ $literal: ... }`. Both matched nothing under the attack. This is a hard compiler invariant, not a guideline.

- **Change streams compose the same predicate — with sharp caveats (the EPIC-L seam).** The compiled read filter, key-prefixed onto `fullDocument.*`, works as a change-stream `$match`: a subscriber received only its permitted insert; the cross-tenant insert was filtered server-side. But: (a) **without `fullDocument: "updateLookup"`, update events carry no `fullDocument`** and are silently dropped by the policy `$match` — permitted updates become invisible (fail-closed but lossy); (b) **delete events never carry `fullDocument`** — policy-filtered deletes are invisible unless the collection enables `changeStreamPreAndPostImages` and the predicate also matches `fullDocumentBeforeChange.*` (verified working after `collMod`, and a cross-tenant delete stayed correctly invisible). EPIC-L must own this contract.

- **Fail-closed DX is a typed error, never a silent empty.** An unknown/unregistered collection → a `PolicyDeniedError` carrying `{ collection, operation, principal }` and an actionable message. The same denial fires when a `$lookup` targets an unregistered collection (fail-closed recursion). The service-role bypass is an explicit context constructor requiring a `reason`, never a flag on user claims — designed directly against Supabase's silent-`permission-denied` footgun.

## Tickets

### RLS-1: Typed policy definition API

- **Priority** P3 | **Estimate** L | **Depends on** EPIC-B (typed CRUD query descriptions), EPIC-C (claims schema via Standard Schema) | **Context** Supabase policies are SQL strings, unchecked until runtime — the community's #1 footgun. PipeSafe's differentiator is compile-time-typed policies over `Schema` + `claims`.
- **Design.** A `definePolicy<Schema, Claims>({ collection, read, write, check })` API in a new `packages/core/src/rls/`. `read`/`write` are `(claims: Claims) => MatchQuery<Schema>` — the returned predicate is validated against the collection schema by the existing `$match` machinery (`ResolveMatchOutput`, `PipeSafeError` operand brands), so a typo'd field or wrong-typed comparison is a compile error. `check` is `(claims, doc: Schema) => boolean | MatchQuery<Schema>` (the `WITH CHECK` half for inserts and `$set` inspection). Policies register on a `Collection` (mirroring EPIC-D `use`) or a standalone registry keyed by collection name. `read` ≈ USING for SELECT, `write` ≈ USING for UPDATE/DELETE, `check` ≈ WITH CHECK for INSERT — the mapping the migration guide (RLS-9) leans on.
- **Acceptance criteria.** Policy predicates fully typed against the collection schema; a typo'd field fires the existing brand; claims type flows from the adapter (RLS-5); registering two policies for one collection is a compile/registration error; `read`/`write`/`check` are independently optional with documented fail-closed defaults when absent (RLS-8).
- **Test plan.** `rls/policy.typeAssertions.ts` — predicate typing, brand firing on bad fields, claims-type propagation; `callSite.typeAssertions.ts` with `@ts-expect-error` pinning rejection of an untyped/SQL-string policy shape.
- **Open questions.** Should `read` be allowed to return a full pipeline fragment (multi-stage) or only a `$match` (v1: `$match` only — matches the spike's compiler and keeps prepend semantics simple)?

### RLS-2: Compilation to find / aggregate / update / delete injection via EPIC-D interceptors

- **Priority** P3 | **Estimate** L | **Depends on** RLS-1, EPIC-D (HOOKS-1 interceptor API, HOOKS-4 ordering)
- **Context.** The interceptor API is the enforcement substrate. Policies compile to filter/`$match` injection that the interceptor onion applies innermost (adjacent to the driver), so no user interceptor can observe or strip them.
- **Design.** A built-in interceptor (registered innermost per HOOKS-4) that, per operation kind: `find`/`findOne` → AND the compiled read filter into the query filter; `aggregate` → prepend `{ $match: readFilter }` and recursively rewrite sub-pipelines (RLS-3); `updateOne`/`updateMany`/`deleteOne`/`deleteMany` → AND the compiled write filter into the query filter (spike: cross-tenant ops report `matchedCount`/`deletedCount` 0 — RLS semantics, not an error); `insertOne`/`insertMany` → evaluate `check` client-side, throw `PolicyDeniedError` on failure. Update `WITH CHECK` inspects `$set` (and `$rename`/pipeline-form updates — flagged) to block escalation like `$set: { tenantId: <other> }`. **All claim values compile to query-context filters (literal); `$expr` is used only when unavoidable and every claim value is wrapped in `$literal` (spike finding — field-path injection is real). Values are always BSON, never string-interpolated.**
- **Acceptance criteria.** Port the spike's read/write/insert scenarios as integration tests: three principals get disjoint correct rows; cross-tenant write reports 0 matched/deleted; cross-tenant insert throws and writes nothing; `$set` escalation rejected; the injected filter cannot be widened by the user filter.
- **Test plan.** vitest on `MongoMemoryReplSet`, mirroring spike sections [1] and [3]; unit tests for the filter-AND composer and the `$expr`-`$literal` wrapper (assert no field-path interpretation).
- **Open questions.** How to handle bulk writes with per-document divergent `check` results — fail the whole batch or partition? (Proposal: fail-closed whole-batch in v1.)

### RLS-3: `$lookup` / `$unionWith` / `$facet` sub-pipeline traversal (the bypass fix)

- **Priority** P3 | **Estimate** L | **Depends on** RLS-2 | **Context** THE spike finding: root-only `$match` prepend is not enforcement — `$lookup` reads any collection wholesale.
- **Design.** A recursive pipeline rewriter (ported directly from the spike's `rewriteStage`): every `$lookup` gets the foreign collection's compiled read policy prepended into its `pipeline` (works for both pipeline form and shorthand `localField`/`foreignField` form — the injected pipeline coexists and ANDs with the equality join, MongoDB ≥5.0); every `$unionWith` is normalized to object form with the target's policy prepended (bare-string form leaks); `$facet` branches are each rewritten; nested `$lookup`-inside-`$lookup` recurse to arbitrary depth. **`$graphLookup` has no sub-pipeline → deny** on any policy-bearing collection; `$out`/`$merge` in a read path → deny (write-through). A `$lookup` whose `from` is an unregistered collection → `PolicyDeniedError` (fail-closed recursion, RLS-8).
- **Acceptance criteria.** Spike section [2] ported: naive prepend leaks tenant-2 secrets via `$lookup`; recursive guard returns only permitted foreign rows; shorthand lookup ANDs policy with join; nested lookups guarded at every depth; bare `$unionWith` closed; `$graphLookup`/`$merge` denied.
- **Test plan.** vitest mirroring spike [2]; a fuzz-ish test asserting every collection-referencing stage in a generated pipeline carries an injected policy `$match` (traversal completeness); regression pin that a new collection-referencing stage without a handler is denied, not silently passed.
- **Open questions.** `$lookup` with `let`/expr correlation — does the injected `$match` interfere with `let` variable scoping? (Spike used plain pipelines; validate the `let` case.) Should `$graphLookup` be closable via a post-filter on `as` rather than a hard deny?

### RLS-4: Change-stream / realtime subscription composition (EPIC-L seam)

- **Priority** P3 | **Estimate** M | **Depends on** RLS-2; composes with **EPIC-L** (realtime) and EPIC-J (change-stream substrate)
- **Context.** The RLS-substitute must extend to realtime, exactly as Supabase's `realtime.list_changes` re-checks RLS per subscriber — but MongoDB lets us push the predicate into the stream (filter-pushdown, not per-event re-query).
- **Design.** Compile the same read policy, key-prefixed onto change-event fields, into a change-stream `$match` (spike's `prefixFilterKeys`). **The delivered contract (spike-verified):** inserts filter cleanly on `fullDocument.*`; updates require `fullDocument: "updateLookup"` or permitted updates are silently dropped; deletes carry no `fullDocument` at all and require `changeStreamPreAndPostImages` + a `fullDocumentBeforeChange.*` predicate branch to be visible (cross-tenant deletes correctly stay invisible). EPIC-L owns the subscription API; this ticket delivers the predicate compiler and the documented contract EPIC-L consumes.
- **Acceptance criteria.** Port spike section [5]: a subscriber receives only permitted inserts; with `updateLookup`, only permitted updates; with pre-images enabled, only permitted deletes; every cross-tenant event is filtered server-side. The contract (updateLookup mandatory, pre-images for deletes) is documented as EPIC-L's requirement.
- **Test plan.** vitest on `MongoMemoryReplSet` mirroring spike [5], including the negative (no-`updateLookup` update dropped) and the `collMod` pre-image path.
- **Open questions.** For a subscriber whose policy references fields not present in a partial update's `fullDocument`, is dropping (fail-closed) acceptable, or does EPIC-L force `updateLookup` universally (latency/cost tradeoff)?

### RLS-5: Auth adapters — JWT claims ingestion (Better Auth / Clerk / custom)

- **Priority** P3 | **Estimate** M | **Depends on** RLS-1, EPIC-C (Standard Schema types the claims) | **Context** Never build auth (market brief §6). Integrate existing issuers; type the claims.
- **Design.** An `AuthAdapter` interface producing a typed `claims` object from a request/token, with first-party adapters for Better Auth, Clerk, and a `customJwt` adapter (verify signature via a provided JWKS/secret, map claims). The claims _schema_ is declared with Standard Schema (EPIC-C) so `claims.sub`, `claims.tenantId`, custom claims are typed end-to-end into the policy predicates (RLS-1). Adapters only _verify and shape_ tokens — issuance, refresh, sessions are the issuer's job. Maps directly to Supabase's `auth.uid()` / JWT-role model but with the claim set typed rather than implicit GUCs.
- **Acceptance criteria.** A verified token produces typed claims that flow into a policy; an unverified/expired token is rejected before any query (fail-closed); the three adapters share one `claims` type surface; custom claims are typed.
- **Test plan.** Unit tests with fixture JWTs per adapter (valid, expired, wrong-signature, missing-claim); type assertion that policy predicates see the declared claims type.
- **Open questions.** Where does verification run in library-interceptor mode (the caller supplies claims) vs gateway mode (the gateway verifies)? (Ties to RLS-7.)

### RLS-6: Fail-closed defaults + service-role escape hatch

- **Priority** P3 | **Estimate** S | **Depends on** RLS-2 | **Context** Designed against Supabase's silent-`permission-denied` footgun and its `security definer` view-bypass leaks.
- **Design.** Default deny: a collection with no registered policy → `PolicyDeniedError` on any principal access (never a silent empty result — spike [6]). Every denial carries `{ collection, operation, principal }` and an actionable message. The service-role bypass is an explicit `serviceContext(reason: string)` constructor (spike) — the _only_ bypass path, never a flag on user claims (which could be forged). A dev-mode warning fires when a service context is used without a `reason` or in a request-scoped path. Explicitly guard against the view-bypass class: any escape hatch (e.g. `collection.raw()` from EPIC-D HOOKS-5) that skips policies is documented as service-privileged and logged.
- **Acceptance criteria.** Spike [6] ported: unknown collection → typed error with full context; `$lookup` to unregistered collection denies the pipeline; service context bypasses and requires a reason; a user-claims `role: "service_role"` value does _not_ bypass (must be the explicit context).
- **Test plan.** vitest mirroring spike [6]; unit test that no code path returns `[]` for a missing policy; assertion that `raw()`/skip hatches are flagged service-privileged.
- **Open questions.** Should fail-closed be overridable to fail-open per collection for public/reference data, or is an explicit `publicRead` policy preset cleaner? (Proposal: `publicRead` preset — never a fail-open toggle.)

### RLS-7: Enforcement-point packaging — library interceptor mode vs standalone gateway mode

- **Priority** P3 | **Estimate** M | **Depends on** RLS-2, RLS-5; **relates to** `../06-architecture-packaging-licensing.md` §5 open decision
- **Context.** PipeSafe today is a library that trusts its caller — there is no server all traffic must traverse (architecture brief §4, §8). RLS enforcement is only as strong as the guarantee that _all_ access goes through the compiled interceptors. Two packaging modes with different trust models.
- **Design.** (1) **Library-interceptor mode:** policies enforced in-process via EPIC-D interceptors; trust boundary is "the app only touches Mongo through PipeSafe collections" (fast, zero-infra, but a direct driver connection bypasses it — documented honestly). (2) **Standalone gateway mode:** a deployable process (HTTP/WebSocket) that verifies JWTs (RLS-5), compiles policies, and is the sole holder of the privileged Mongo credential — the App-Services-replacement shape, and the enforcement point that makes RLS a real security boundary. This ticket scopes the shared policy-compiler core both modes reuse and defers transport/routing/hosting to the BaaS epic. Licensing (Apache core vs ELv2 gateway) and the CLI-license precedent are the `plan/06` §5 decision this ticket surfaces, not resolves.
- **Acceptance criteria.** The policy compiler is transport-agnostic and shared; library mode documented with its trust caveat; a gateway-mode design note specifies the credential-isolation invariant (gateway holds the only privileged connection). No transport implementation in v1 — this is the packaging decision + shared core.
- **Test plan.** Integration test that the same compiled policy produces identical injected pipelines whether invoked via interceptor or via a stub gateway handler (mode parity).
- **Open questions.** Which package owns the gateway (new `@pipesafe/gateway` under ELv2, per the manifold precedent)? Resolve with `plan/06` §5 before shipping transport.

### RLS-8: Policy-denial observability — events into the EPIC-G event log

- **Priority** P3 | **Estimate** S | **Depends on** RLS-6, EPIC-G (event log) | **Context** Supabase's RLS debugging is the "nine log tabs, manual timestamp correlation" nightmare (market brief §3). Observability is a differentiator, not an afterthought.
- **Design.** Every `PolicyDeniedError` (and every service-context bypass) emits a structured event into the `_manifold` append-only log (EPIC-G): `{ v, type: "policy.denied" | "policy.bypass", collection, operation, principal, reason, ts }`. Denials are queryable and aggregatable (denial rate per principal/collection — an abuse signal). No document _contents_ are logged (only keys/predicate shape) to avoid leaking the data the policy protects. A dev-mode explainer surfaces _which_ policy clause denied (the actionable "why", unlike Supabase's opaque `permission denied`).
- **Acceptance criteria.** A denied read/write/insert produces exactly one event with full context; a service bypass is logged; no protected field values appear in the event; the event conforms to EPIC-G's versioned union.
- **Test plan.** vitest asserting event emission and shape on each denial path; a redaction test that no seeded secret value appears in any emitted event.
- **Open questions.** Sampling for high-volume denial storms (a scanning attacker) — cap/aggregate, or log every one? (Proposal: aggregate with a count + first/last exemplar.)

### RLS-9: Migration-from-Supabase guide + Supabase-idiom mapping

- **Priority** P3 | **Estimate** M | **Depends on** RLS-1, RLS-2, RLS-6 | **Context** This epic's reason to exist is unblocking the Supabase→Atlas migration; the guide is the deliverable that makes the unlock legible to a migrating team.
- **Design.** A mapping doc + worked example translating Supabase RLS idioms to PipeSafe policies: `USING (auth.uid() = user_id)` → `read: (c) => ({ ownerId: c.sub })`; `WITH CHECK (...)` → `check`; per-operation policies (`FOR SELECT/INSERT/UPDATE/DELETE`) → `read`/`check`/`write`; `auth.uid()`/`auth.jwt()` → typed `claims` (RLS-5); role-based policies (`anon`/`authenticated`/`service_role`) → claim-driven predicates + `serviceContext`. Call out the traps PipeSafe designs _against_: the `security definer` view bypass (PipeSafe has no view-bypass because `$lookup`/`$unionWith` are recursively guarded — RLS-3), and silent `permission denied` (PipeSafe throws typed errors with the denying clause — RLS-6/RLS-8). Include a checklist for auditing that all access routes through the enforcement point (RLS-7).
- **Acceptance criteria.** Every common Supabase RLS pattern has a mapped PipeSafe equivalent with a runnable snippet; the two footguns (view bypass, silent denial) are explicitly contrasted; the guide references the enforcement-point trust model honestly.
- **Test plan.** The worked-example policies compile and pass RLS-10 assert-visibility fixtures (the guide's examples are executable, not prose-only).
- **Open questions.** Ship as docs-site content or an in-repo `examples/` file first? (Proposal: `examples/` executable first, docs later.)

### RLS-10: Policy testing utilities — assert-visibility fixtures (EPIC-H tie-in)

- **Priority** P3 | **Estimate** M | **Depends on** RLS-2, RLS-3, EPIC-H (testing framework) | **Context** Untested policies are the Supabase failure mode. Policy tests must be first-class, reusing EPIC-H's data-test-as-pipeline harness.
- **Design.** An `assertVisibility({ collection, principal, canSee, cannotSee })` fixture utility: given seeded data and a principal's claims, assert the exact permitted row set (spike's `ids()` comparison pattern). Built on EPIC-H's typed fixtures (`$documents` seeding) and data-tests-as-DAG-nodes, so visibility assertions run in the same `build` as model/data tests. Ships presets for the common shapes: tenant-isolation, owner-only, role-tiered, and the `$lookup`-bypass regression (assert a crafted malicious pipeline sees nothing cross-tenant — RLS-3's guarantee, pinned as a test). Negative assertions (`cannotSee`) are as important as positive.
- **Acceptance criteria.** A visibility fixture fails when a policy is too permissive (leaks a row into `cannotSee`) _or_ too restrictive (drops a `canSee` row); the `$lookup`-bypass regression fixture fails if traversal (RLS-3) is disabled; fixtures run as EPIC-H DAG test nodes.
- **Test plan.** Self-describing (the ticket _is_ a test harness); a meta-test that a deliberately-broken policy is caught by the fixture; integration with EPIC-H's `build` runner.
- **Open questions.** Should visibility fixtures auto-generate adversarial pipelines (every collection-referencing stage) to prove traversal completeness, or is a curated regression set enough for v1? (Proposal: curated set + a `$lookup`/`$unionWith` adversarial pair, expand later.)
